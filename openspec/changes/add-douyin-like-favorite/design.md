## Context

当前仓库已经具备两层能力：
- `mcp-server/` 通过 `DouyinUploader` + `index.ts` 对外暴露 MCP 工具
- `skills/douyin/` 通过独立脚本把同样的浏览器自动化能力暴露为 skill 命令

现有自动化能力集中在登录、视频上传、图文上传，底层已经具备 Cookie 持久化、浏览器启动、登录态校验、页面导航等基础设施。新的需求不是发布内容，而是对一个给定的抖音内容链接执行互动操作，并且要求同时覆盖 MCP 与 skill。

用户给出的输入链接是 `https://v.douyin.com/.../` 这种短链接格式。这个约束意味着实现链路必须先把短链接在真实浏览器上下文中解析到最终内容页，然后才能在内容页判断并执行点赞、收藏动作。

## Goals / Non-Goals

**Goals:**
- 新增一个可复用的内容互动方法，对指定抖音内容执行“点赞 + 收藏”
- 同时新增 MCP 工具入口和 skill 命令入口，保持两层能力一致
- 支持用户直接传入抖音短链接，由系统自动完成跳转和解析
- 同时适配抖音视频内容页与图文内容页
- 返回结构化结果，明确最终解析地址、点赞状态、收藏状态与失败原因

**Non-Goals:**
- 不支持只点赞或只收藏的半动作模式
- 不支持批量处理多个内容链接
- 不支持评论、转发、关注等其他互动动作
- 不修改现有上传、登录、Cookie 管理的行为
- 不引入新的外部服务或额外依赖

## Decisions

### 1. 继续复用 `DouyinUploader` 作为互动能力承载类

**决策**：在现有 `DouyinUploader` 中新增内容互动公共方法，而不是新建第二个自动化核心类。

**理由**：当前浏览器启动、Cookie 读取、登录态校验、异常收敛都已经集中在 `DouyinUploader` 内。点赞收藏与上传虽然业务动作不同，但依赖的会话和自动化基础完全一致，继续复用是最短路径。

**备选方案**：新建 `DouyinEngagementClient`。
放弃原因：会复制浏览器与 Cookie 逻辑，增加维护面，不符合最短路径实现。

### 2. 使用浏览器导航解析短链接，而不是先做 HTTP 层重定向解析

**决策**：直接让 Puppeteer 打开用户提供的短链接，等待页面跳转到最终内容页，再基于最终页面执行后续动作。

**理由**：短链接解析与最终可互动状态依赖真实浏览器会话。浏览器内解析可以天然复用 Cookie、重定向、脚本渲染和登录态，不需要额外维护请求头、反爬策略和页面差异。

**备选方案**：先用 `fetch`/HTTP 请求解析短链，再把结果 URL 交给浏览器。
放弃原因：会把“链接解析”和“登录态页面交互”拆成两套上下文，增加不一致风险。

### 3. 点赞和收藏按“目标状态”执行，而不是无条件点击

**决策**：进入内容页后先检测点赞和收藏当前状态，仅在目标状态未达成时点击；如果内容已经点赞或已收藏，直接视为该动作成功。

**理由**：点赞和收藏按钮通常是切换式按钮，无条件点击会把已完成状态反向取消，导致结果和需求相反。需求本质是“确保内容已点赞且已收藏”，不是“执行一次点击”。

**备选方案**：固定点击一次点赞按钮和收藏按钮。
放弃原因：逻辑不正确，无法满足最终状态目标。

### 4. 为 skill 新增独立脚本入口，而不是复用 `upload.js` 或 `manage.js`

**决策**：在 `skills/douyin/scripts/` 下新增独立互动脚本，并在 skill 包里注册新的命令入口。

**理由**：上传和互动是两类不同语义。把点赞收藏塞进 `upload.js` 会让参数模型混乱，把它塞进 `manage.js` 会污染账号管理边界。独立入口最清晰，也便于后续文档说明。

**备选方案**：在 `upload.js` 中通过 `--type engage` 复用。
放弃原因：会把“内容发布”和“内容互动”混成一个脚本，不利于参数校验和可维护性。

### 5. 结果对象同时包含动作结果与解析结果

**决策**：返回结构中同时包含输入链接、解析后的最终内容页地址、点赞结果、收藏结果和错误信息。

**理由**：短链接是输入，真正执行动作的是跳转后的最终页面。若不返回解析结果，调用方无法判断是否对正确内容完成了操作。

**备选方案**：仅返回 `success: boolean`。
放弃原因：信息不足，无法支撑排障和结果确认。

## Risks / Trade-offs

- **[页面结构变动]** → 抖音内容页可能改版，导致点赞或收藏选择器失效。缓解：使用“文本特征 + 可点击元素 + 状态属性”组合判断，而不是单一类名。
- **[视频与图文页面差异]** → 两类内容页的按钮布局可能不同。缓解：先抽象“查找点赞按钮/收藏按钮”的通用策略，再分别覆盖两类页面的候选选择器。
- **[已登录但互动受限]** → 某些页面可能弹出登录确认层或权限弹层。缓解：在正式动作前做一次登录态与遮挡层检查，必要时返回明确错误而不是盲点。
- **[短链目标不可访问]** → 链接可能失效、内容删除或私密。缓解：在跳转完成后校验最终页面是否可交互，不可交互时返回结构化失败。

## Migration Plan

1. 在 TypeScript 侧新增互动参数类型、返回类型、浏览器自动化方法与 MCP 工具注册
2. 在 skill 侧同步新增同名能力与独立命令脚本
3. 更新 skill 文档，说明新命令用途、参数与使用示例
4. 通过构建或最小自动化测试验证两层入口都能完成参数校验和结果返回

本变更为纯增量，不涉及数据迁移，也不需要回滚脚本。若后续需要回滚，只需移除新增工具、脚本与互动方法。

## Open Questions

无。当前需求边界明确：对指定抖音内容链接执行点赞+收藏，覆盖 MCP 与 skill，两者都不需要额外业务分支。
