## Context

当前 Douyin MCP Server 通过 Puppeteer 浏览器自动化实现视频上传功能。核心架构为：
- `DouyinUploader` 类封装所有浏览器自动化逻辑（登录、Cookie 管理、视频上传）
- `DouyinMCPServer` 类作为 MCP 协议适配层，将 `DouyinUploader` 的能力暴露为 MCP 工具
- 使用 Zod 进行参数校验，Cookie 持久化实现跨会话登录

抖音创作者平台的图文发布入口与视频发布入口不同，图文通过 `https://creator.douyin.com/creator-micro/content/post/image` 页面进行发布。图文发布流程包括：上传图片 → 填写标题和描述 → 添加话题 → 选择背景音乐 → 发布。

## Goals / Non-Goals

**Goals:**
- 新增图文作品上传能力，支持标题、描述、图片、话题、背景音乐
- 复用现有基础设施（浏览器管理、Cookie 管理、登录态、短信验证）
- 保持与视频上传一致的代码风格和错误处理模式
- 提供与视频上传工具对等的 MCP 工具接口

**Non-Goals:**
- 不改造现有视频上传流程
- 不支持图文与视频混合发布
- 不支持编辑已发布的作品
- 不支持从 URL 下载图片（仅支持本地文件路径）
- 不支持定时发布功能（仅立即发布或保存草稿）

## Decisions

### 1. 图文上传方法独立于视频上传

**决策**：在 `DouyinUploader` 中新增独立的 `uploadImages()` 方法，而非扩展 `uploadVideo()`。

**理由**：图文和视频的上传页面 URL 不同、表单元素不同、处理流程不同。共用方法会增加复杂度和维护负担，独立方法更清晰。

**备选方案**：统一为 `upload()` 方法通过参数区分类型 —— 放弃，因为两种上传流程差异较大，强行统一会导致大量条件分支。

### 2. 图片文件验证策略

**决策**：在上传前验证所有图片文件存在性和格式，任一校验失败则整体拒绝。

**理由**：提前失败（fail-fast）比上传中途失败体验更好，避免浪费用户时间。

**备选方案**：逐个上传并跳过无效图片 —— 放弃，因为部分成功状态难以处理，且可能导致不符合预期的发布结果。

### 3. 背景音乐实现方式

**决策**：通过搜索关键词匹配的方式选择音乐，用户提供音乐名称关键词，自动搜索并选择第一个匹配结果。

**理由**：抖音平台的音乐选择基于搜索，不支持外部音乐文件上传。通过关键词搜索并自动选择第一个结果是最直接可靠的方式。

**备选方案**：支持精确音乐 ID —— 放弃，因为音乐 ID 对用户不友好，且获取 ID 需要额外的 API 或页面操作。

### 4. 复用发布和短信验证流程

**决策**：将视频上传中的发布按钮点击和短信验证逻辑提取为私有方法，在视频和图文上传中共用。

**理由**：发布和验证流程与内容类型无关，是平台级行为，代码复用可减少维护成本。

## Risks / Trade-offs

- **[页面结构变化]** → 抖音创作者平台 UI 可能变动导致选择器失效。缓解：使用多种选择器策略（文本匹配、CSS 类名、DOM 结构），与视频上传保持相同的容错模式。
- **[音乐搜索不稳定]** → 背景音乐搜索结果可能变化或无结果。缓解：音乐为可选功能，搜索失败不阻塞发布流程，返回警告信息。
- **[图片数量限制]** → 抖音平台对单个图文作品的图片数量有限制（通常最多 35 张）。缓解：在参数校验层提前检查数量上限。
- **[上传耗时]** → 多张大尺寸图片上传可能耗时较长。缓解：根据图片数量和总大小动态计算等待时间。
